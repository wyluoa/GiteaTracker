1. 適用對象

這份指南寫給需要部署、維運、進階操作系統的同仁，涵蓋：

  • Super user：後台管理、使用者審核、Excel 匯入、關單確認等日常操作。
  • Maintainer：負責在公司內部伺服器（f12titan）部署與升級。
  • 接手的開發者：理解資料模型、狀態守衛規則、擴充 node 或重新部署。
一般使用者請看另一份 docs/user_guide.html 已涵蓋絕大多數日常操作，不需要這份的技術細節。

3. 首次部署

3.1 前置條件

  • Linux 主機（公司 f12titan），不需要 sudo 權限。
  • Python 3.12 已安裝在 user-space 或公司 module 中。
  • 可寫入家目錄下任一路徑（放 DB 和附件）。

3.2 建立 virtualenv 與安裝依賴

cd ~/GiteaTracker
python3 -m venv venv
source venv/bin/activate
pip install -r requirements.txt

3.3 設定環境變數

複製 .env.example 為 .env，填入： 

FLASK_ENV=production
SECRET_KEY=<random 32+ chars>
DB_PATH=data/tracker.db
ATTACH_DIR=data/attach
SCRIPT_NAME=/GiteaTracker        # reverse proxy 路徑前綴
SMTP_HOST=smtp.company.internal
SMTP_PORT=25
MAIL_SENDER=gitea-tracker@company.internal
ADMIN_USERNAME=wy                # 首次 seed 的 super user
ADMIN_EMAIL=wy@company.internal
ADMIN_PASSWORD=<強密碼>
SECRET_KEY 請產生隨機字串 python -c 'import secrets; print(secrets.token_urlsafe(48))' 不要用範例值。

3.4 初始化資料庫

python init_db.py

這個 script 會：執行 app/schema.sql 建表、插入 10 個預設 node（A10…MtM）、建立 ADMIN_USERNAME 指定的 super user 帳號（狀態 active）。

3.5 (選) 匯入現有 Excel

python import_from_excel.py path/to/control_table.xlsx

會把 Ongoing / Closed 兩個 sheet 都讀進來；每題建立對應 issue_node_states row。

4. 啟動 / 停止 / 重啟

4.1 啟動（foreground 測試）

venv/bin/python main.py
# 預設 port 在 .env 裡設為 9987（舊版 README 寫的 5000 是過時的）

4.2 背景執行（無 systemd 的環境）

# 用 nohup 掛背景
nohup python main.py > logs/app.log 2>&1 &
echo $! > app.pid

# 停止
kill $(cat app.pid)

# 重啟（deploy 新版後）
kill $(cat app.pid) && \
  nohup python main.py > logs/app.log 2>&1 & echo $! > app.pid

4.3 用 csh 的環境

公司有些機器預設 csh。deploy/ 下有 csh 版啟動 script：

source deploy/start.csh

4.4 Reverse proxy 前綴

若外層 Traefik / Nginx 會把 /GiteaTracker/xxx 剝掉 prefix 再 forward， .env 的 SCRIPT_NAME=/GiteaTracker 會讓 Flask 的 url_for() 產生帶前綴的 URL。若沒有反向代理，設成空字串即可。

5.1 關鍵 cache 欄位

欄位意義 / 何時更新
issues.all_nodes_done 該題所有 node 都是 done 或 unneeded 時為 1。 每次 cell 編輯後 issue.refresh_cache() 重算。
issues.latest_update_at 最後一次任一 cell 被更新的時間。用於 Stale Topics 分析。
issues.updated_at 2026-04-21 只在 meta 欄位（topic/owner/jira/uat_path/icv/week/group_label）變動時 bump，不因 cell 狀態改動而 bump（refresh_cache 已拿掉對它的更新）。這樣 Owner/JIRA/Path 的 per-column highlight 才不會被 cell traffic 污染。
issues.topic_updated_at
issues.owner_updated_at
issues.jira_updated_at
issues.uat_path_updated_at 2026-04-21		 每個 meta 欄位各自的 timestamp，由 issue_model.update_issue() 依 FIELD_TO_TS 對應表自動 bump。Tracker 以此判斷「這一欄是否晚於使用者 last_viewed_at」，只 highlight 真的被改的那欄。
issues.pending_close Excel 匯入時 Ongoing 分頁 Status=Closed 會被設為 1，等 admin 人工確認。
issues.is_deleted Soft delete。所有 query 都會 filter 這個。Admin 有 restore 功能。
users.last_viewed_at 2026-04-21 版本差異 highlight 的 per-user 基準點。新使用者在 user_model.create_user() 建立當下自動設為 now()（option B），避免第一次登入看到一整片黃底。舊使用者若 NULL 需手動按一次「標記已讀」。
schema_version.version 2026-04-21 Migration runner 用來追蹤已套用版本。每次 migrate.py 成功套用一個 migration 就寫一筆。

6. Admin 後台詳解

導覽列右側點 Admin 進入（super user 限定）。左側側邊選單區分為十個區塊：

6.1 Pending Users

列出狀態為 pending 的註冊申請。兩個動作：

  • 啟用：改 status → active，使用者即可登入。
  • 拒絕：改 status → disabled，不可登入；日後可改回 active。

啟用時可選勾「寄通知信」，前提是 SMTP 設好。

6.2 Users

管理所有使用者。欄位：

欄位說明
username / email / display_name基本資料。
Statuspending / active / disabled。可點下拉修改。
Super勾起來 = super user，擁有所有權限。
Manager 2026-04 勾起來 = manager。相對 super 更窄，僅可額外設 Unneeded；其他編輯權限依 group 綁。
Group一人可屬多組。控制該使用者能編輯哪些 node 的 cell。
Reset Password點了可直接改使用者密碼（不寄信）。

6.3 Groups

Group 是「帳號 ↔ Node 編輯權限」的中介。UI 示意： 

┌ Group: A10_team ─────────────────────────┐
│  Members: WY, Alice, Bob                 │
│  Nodes:   [A10] [A12]                    │
│                                          │
│  + Add Node   × Remove Node              │
└──────────────────────────────────────────┘

建立步驟：

  1. Admin → Groups → 新增群組，輸入名稱。
  2. 進入群組頁，勾選此群組負責的 node。
  3. 回 Users 頁，把人加入群組。

6.4 Nodes

  • 新增 / 重新命名 / 調整排序 / 停用 node。
  • 停用的 node 不會出現在 Tracker 欄位，但已有的 issue_node_states row 仍保留，重新啟用後會恢復顯示。
  • 排序用拖拉或改 sort_order 數字。
不要刪除 node 刪除會破壞歷史 timeline 的 FK。一律用「停用」取代刪除。

6.5 Red Line

設定 red_line_year + red_line_week。儲存後：

  • Tracker 在對應週次之間畫紅虛線。
  • Red Line 之前仍為 UAT / TBD 的 cell 變紅警示。
  • Dashboard 的 Per-Node Stats 會把「紅線前」和「全部」分開統計。

6.6 Gitea URL Mapping

每筆規則有 prefix 和 url_template： 

prefix       url_template
─────        ──────────────────────────────────────
(空)         http://gitea/QC_requests/issues/{number}
P            http://gitea/perl_scripts/issues/{number}
CN           http://gitea/CN_TEAM/issues/{number}
SNPS         http://gitea/Synopsys/issues/{number}

解析順序：先看 display_number 是否以某 prefix 開頭（取最長匹配）； 如果都沒中，且 display_number 是純數字，才套用 prefix 為空的預設規則。

6.7 Column Width

CSS variables --col-topic-min-width 和 --col-path-min-width 由這裡設定（單位 px）。 建議：Topic 280、Path 220。Path 有多行路徑時可拉大到 400-500。

6.8 Trend Data 2026-04-29

Dashboard 的 Topic Cumulative Trend 與 Closing Rate Trend 兩張圖的單一資料源。每週手動輸入該週當下的 UAT / TBD / Dev / Close 題數，圖才會出現該週點。

  • 沒有 fallback：以前曾有「沒填就從現有 issues 算」的 fallback（get_dashboard_trends()）， 已於 2026-04-29 移除。原因：歷史 cumulative 不能從現況回推，那會把每題畫到首次提出週、無法表達歷史快照。
  • 本週快照：Dashboard 上方 Weekly summary 卡會顯示 current_phase_snapshot()（app/models/issue.py）算出的當下分類結果， Super User 可一鍵點「填入 Trend Data」帶 query string （prefill_year / prefill_week / uat / tbd / dev / close） 跳轉到本頁，新列被自動 prefill。
  • 分類邏輯（每題只歸一類，priority Close > UAT > TBD > Dev）：
    • Close = issues.status = 'closed'
    • UAT = 任一 node state = 'uat'（不含 uat_done，因 UAT 類別語意是「尚未測試」）
    • TBD = 任一 node state = 'tbd'
    • Dev = 其餘（含 uat_done、空白、全 done/unneeded 但未關單、on_hold 無 uat/tbd）
  • 歷史資料：2026-04-29 之前填過的舊週數字是按舊定義（UAT 含 uat_done）填的， 不會自動回填。如要對齊定義，到本頁手動把對應週的 UAT 移到 Dev。

6.9 SMTP

設定寄信所需：

欄位說明
SMTP Host / Port公司 mail relay，通常 port 25 或 587。
Mail SenderFrom address（要是公司核准的發信人）。
TLS / Auth若 relay 要認證填帳密。
測試寄信表單下方可寄測試信驗證。
也可改回 ddi_api.pl 內部 perl API 可用時速度更快、不需 SMTP 認證。切換點在 app/mail.py。

6.10 Audit Log

所有改動 users / groups / nodes / red line / settings / issues 的動作都會記在 audit_log。Admin 頁面可搜尋：actor、action、target_type、時間區間。 Payload 欄是 JSON，詳述前後值。

7. Excel 匯入流程

7.1 格式要求

  • 工作簿必須有 Ongoing 與 Closed 兩張 sheet（名字可再加副標，系統以開頭為準）。
  • 首列是欄位標題，可容忍中英文：# / 題號 / Topic / Owner / JIRA No. / ICV / UAT Path / Status。
  • Node 欄以 node display_name 為欄頭（A10, A12, N2…）。
  • 每格的 state + check-in + note 可以各種形式書寫，app/excel.py::parse_cell 會 best-effort 解析。

7.2 邊界情況（已內建處理）

狀況行為
同一 Excel 欄頭 N2/A16 被合在一起 拆成兩個 node 各自插一筆 state。
欄頭寫 JIRA No.視為 jira_ticket。
週次列（空 row、只有 wkXXX 字串）作為該 group 的 current week，後續題目 inherit。
單獨 group label 列（例：強身健體系列） 作為後續題目的 group_label 直到下個 label 或週次切換。
Ongoing sheet Status = Closed 不自動關，設 pending_close = 1 等 admin 人工確認。
Closed sheet直接進 status=closed，不走 pending_close。
題號重覆（相同 display_number 在不同列）匯入時 warning，第一筆為準。
Owner / Path 空白允許 null，不擋。

7.3 透過 Web UI 更新（Diff Preview）

  1. Admin > Excel Update 上傳新版 .xlsx。
  2. 系統跑 dry-run，呈現：新增題目 / 現有題目欄位差異 / pending_close 建議關題清單。
  3. 逐項勾選要套用的變更（細到「題 #156 的 Topic 改成 …」這種 granularity）。
  4. 確認後一次套用，所有改動進 audit log，timeline 補一筆 Imported via Excel upload。

7.4 CLI 匯入

python import_from_excel.py control_table.xlsx

適合首次 seed。不走 diff preview，直接全量寫入（若題號已存在會跳過並警告）。

當某 cell 目前狀態是該使用者沒權限改到的（例如 general user 看到 done）， side panel 的 state 下拉會 disabled 並附註此標籤。避免使用者誤以為可改。

9. 關單 / Reopen / pending_close

9.1 三種關單路徑

情境觸發UI
正常完成 所有 node 為 Done / Unneeded → all_nodes_done=1 題號旁綠色 ✓關 按鈕。
Excel 匯入建議關題 Ongoing sheet Status=Closed → pending_close=1 題號旁黃色 ✓關? 按鈕；Dashboard 獨立卡片。
手動強制關單 Admin 無視 all_nodes_done，直接按 Close 即使有 node 還沒 Done 也可關（記 audit log）。

9.2 Reopen 行為

  • 從 Closed 頁面點 Reopen，狀態 → ongoing。
  • closed_at / closed_note 清空。
  • Timeline 補一筆 reopen entry（含原因）。
  • 所有原本的 node state 和 timeline 都保留。

9.3 soft delete vs close

兩者不同
關單：題目已完成，狀態 closed，顯示在 Closed 頁，可 reopen。
Soft delete：題目根本不該存在（誤建或重覆），is_deleted=1，Admin 才看得到並可 restore。

10. 郵件通知

系統會在下列情況寄信（前提 SMTP 設定好）：

事件收件人
新使用者申請所有 super user。
帳號啟用 / 拒絕通知申請者。
忘記密碼 → 重設連結申請者（連結 24 小時有效）。

app/mail.py 有 Jinja 模板，如需客製改 HTML 內容直接編輯該檔。

11. 備份與還原

11.1 備份

整個系統就三樣東西要備：

  • data/tracker.db：所有資料。
  • data/attach/：timeline 附件檔。
  • .env：config（含 SECRET_KEY；遺失會讓舊 session 失效）。

排程（可寫 cronjob）：

# 每天 2am 打包一份到家目錄 backup/
0 2 * * * cd ~/GiteaTracker && \
  tar czf ~/backup/tracker-$(date +\%Y\%m\%d).tar.gz data/ .env

保留 7-14 份滾動覆蓋足夠（SQLite 壓縮後約 10-20 MB）。

11.2 還原

  1. 停服務：kill $(cat app.pid)。
  2. 解壓到 ~/GiteaTracker/：tar xzf ~/backup/tracker-20260418.tar.gz。
  3. 確認 data/tracker.db 大小合理。
  4. 啟服務：nohup python main.py &。

11.3 「我想確認資料沒壞」

# 用 Python（在沒裝 sqlite3 CLI 的機器也可以跑）
venv/bin/python -c "import sqlite3; c=sqlite3.connect('data/gitea_tracker.db'); print(c.execute('PRAGMA integrity_check').fetchone()[0])"
# 回傳 ok 即資料庫健康

12.1 一鍵升級（正式機）

  4. migrate.py --dry-run — 預覽要套哪些 migration（顯示給操作者看）
  5. migrate.py — 實際套用
  6. start.sh — 啟動

12.3 Migration 系統架構 新

元件用途
migrate.py Runner。掃 migrations/、比對 schema_version 表、依序跑沒套過的 migration。支援 --list / --dry-run。
migrations/001_*.py
migrations/002_*.py
...		 每個 migration 一個檔，命名 NNN_snake_name.py。內容三個必備欄位：SCHEMA_VERSION、DESCRIPTION、def up(conn)。
app/schema.sql Fresh DB 的完整 schema。加欄位時 schema.sql 和 migration 都要同步更新 — schema.sql 給新 DB 一次到位，migration 給既有 DB 補。
schema_version 表 記錄每次成功套用的 migration version + applied_at + description。
init_db.py 新建 DB 時跑：執行 schema.sql 之後自動呼叫 migrate.run()，讓 fresh DB 和既有 DB 走同一條路。

12.4 新增 Migration（開發者）

  1. 看 migrations/ 目錄，下一個三位數編號（例如 003）。
  2. 建檔 migrations/<NNN>_<short_name>.py。模板見 deploy/MIGRATION_SOP.md 或直接參考 001_issue_meta_timestamps.py（加欄位 + 回填）/ 002_baseline_last_viewed_at.py（純資料更新）。
  3. 同步更新 app/schema.sql（若涉及欄位 / 表結構）。
  4. 本機測試：
    cp data/gitea_tracker.db /tmp/migtest.db
    DB_PATH=/tmp/migtest.db venv/bin/python migrate.py
    DB_PATH=/tmp/migtest.db venv/bin/python migrate.py # 第二次必須 no-op
  5. Fresh DB 測試：
    rm -f /tmp/fresh.db
    DB_PATH=/tmp/fresh.db venv/bin/python init_db.py
    DB_PATH=/tmp/fresh.db venv/bin/python migrate.py --list # 全部 applied
  6. Commit 分開：migrate(NNN): short description。

12.5 冪等性四項檢查

每個 migration 的 up() 寫完檢查四項，都要「是」：

  • 加欄位前有 PRAGMA table_info 檢查？
  • 加表用 CREATE TABLE IF NOT EXISTS？
  • 加 index 用 CREATE INDEX IF NOT EXISTS？
  • UPDATE / INSERT 有條件避免覆蓋已處理的 row（例如 WHERE col IS NULL）？
禁區
  • Migration 內 不要 conn.commit() — runner 會統一處理，自己 commit 會搞壞 rollback 能力。
  • Migration 絕對不動 data/attachments/ 檔案或 attachments.stored_path 欄位 — 附件搬家另寫 one-shot script。
  • 不做 SQLite 不擅長的事：DROP COLUMN、改欄位型別 / constraint。真的要結構性改動就「新欄位 + 回填 + 切換」。
  • 不在 up() 裡 import app code — 只用 sqlite3 和 stdlib，避免啟動副作用。

12.6 升版前先看這次的專屬說明

每次正式升版都會放一份檔案在 deploy/releases/YYYY-MM-DD-*.md，列這次 額外要做的事（例如 pip install、行為改變、使用者要硬重整等）。 migrate.sh 不會自動 pip install，所以若 requirements.txt 有動，要手動分步（看 deploy/MIGRATION_SOP.md 「手動分步流程」節）。

歷史註記：舊版 app/db.py::_run_migrations（每次啟動跑的迷你 migration） 已於 migration 008 合併並移除。所有 schema 變動一律走 migrations/。

12.7 Rollback

升版失敗時：

  1. ./deploy/stop.sh || true
  2. ls -lt backups/ 找最新備份
  3. ./deploy/restore.sh backups/gitea_tracker_<ts>.db backups/attachments_<ts>.tar.gz
  4. 視情況 git checkout <前一個 commit>，再 ./deploy/start.sh

Restore 後 schema_version 也會回到升版前狀態。下次 migrate.py 會把 pending 重跑一遍，因 migration 是冪等的（應該都是）再跑不會壞。若某個 migration 有 bug 導致 rollback，先修那個 migration 的 code 再升。

13. 快取與部署更新 2026-04

13.1 行為概述

  • HTML 回應 header：Cache-Control: no-cache, must-revalidate。 每次 request 都會回 server 驗。
  • 靜態檔（CSS / image）：Cache-Control: public, max-age=604800（7 天）。
  • app.css 以 ?v=<mtime> fingerprint。 mtime 在 Flask 啟動時讀一次，注入到所有 template。

13.2 部署後要做的動作

  • 後端：git pull, 重啟 app。新 app.css 的 mtime 會讓 ?v= 自動變。
  • 使用者：一次硬刷（Ctrl+Shift+R） 清掉之前沒有版號的舊 HTML 快取。之後自動接管。

13.3 為何需要

在 2026-04 之前，公司代理把 app.css 快取一週不重驗， 導致 dark / gray mode 新增的 CSS rule 公司 Chrome 看不到。加了 fingerprint 後這類問題不會再發生。

14. 常見問題排除

14.1 Dark / Gray mode 沒生效

→ 硬刷一次（Ctrl+Shift+R）。若還是無效，檢查瀏覽器是否有擴充套件注入 CSS filter（極少見）。

14.2 登入後一直被踢出

  • 檢查 SECRET_KEY 是否有改過 → 改過的話舊 session 失效，登出再登入即可。
  • 檢查 SESSION_HOURS（預設 24）是否設太短。

14.3 Excel 匯入跳 warning「題號重覆」

表示同個 display_number 出現在不同列。建議修 Excel、去除重覆，再匯一次。

14.4 SMTP 測試寄信失敗

  • TLS port 要 587、非 TLS 通常 25。
  • 寄件人 (sender) 是否在公司白名單？
  • relay 是否允許此伺服器 IP？

14.5 凍結欄位標題不黏

確認 .tracker-table-wrap 的 CSS 是 overflow: clip（非 hidden）。 overflow: hidden 會讓元素成為 scroll container 讓 position: sticky 失效。

14.6 題目一直顯示 Ready to Close 但 node 還沒全部 Done

邊界狀況：新題目用「+ 新增題目」建立時不會預插 issue_node_states， 第一個 node 設 Done 後 MIN 聚合會誤判為全部完成。建議用 Excel 匯入或先把所有 node 都設個 state 再關單。修正中。

17. CSRF / 測試 / Observability 2026-04-24 新增

17.1 CSRF 保護（app/csrf.py）

所有 POST/PUT/PATCH/DELETE 請求必須帶 token（form 欄位 csrf_token 或 header X-CSRFToken）。沒帶或對不上就 403。

  • Token 存在 session["_csrf"]，由 secrets.token_urlsafe(32) 產生，第一次讀到才生。
  • 模板用 {{ csrf_token() }}（context processor 注入）。Python 代碼用 app.csrf.get_or_create_token()。
  • base.html 有 <meta name="csrf-token"> 和 fetch/HTMX wrapper，自動把 token 注入到所有 fetch() 和 HTMX POST。
  • 豁免 endpoint：auth.login / auth.register / auth.forgot_password / auth.reset_password / main.healthz（見 EXEMPT_ENDPOINTS）。
  • 要額外豁免某個 route 加 @csrf_exempt 裝飾器（謹慎使用）。
升版後使用者必須硬重整 舊快取的 HTML 沒有 csrf-token meta。升版完通知：Ctrl+Shift+R。

17.2 測試套件（tests/）

venv/bin/pytest -q    # ~20s，全綠代表 infra + model 核心行為都沒壞
  • 每個測試拿到自己的 tmp SQLite（tests/conftest.py），schema.sql + migrate.py 重跑——跟 production 同一個初始化路徑。
  • Fixtures：super_user / manager_user / editor_user、make_user、sample_issue、old_issue（back-date created_at）、login_as。
  • Migration 冪等性測試是 parametrized，新加的 migration 會自動被測——但有 bug 的 migration 也會自動 fail。
  • 用 WTF_CSRF_ENABLED = False 讓大部分測試繞過 CSRF；test_csrf.py 另起一個 fixture 專門測 CSRF。

17.3 結構化錯誤 log（app/errors.py + /admin/errors）

未處理的例外會寫一筆 JSON 到 logs/errors.jsonl：時間、路徑、操作者、 錯誤類型、完整 traceback、remote_addr。

  • 寫入是 best-effort — 若 log 寫失敗會被吞掉，避免把 500 變成第二個 500。
  • Admin 介面在 /admin/errors（super user only），tails 最新 N 筆，traceback 用 <details> 收合。
  • 沒做自動 rotate。log 變大後手動： 
    cd ~/GiteaTracker/logs
mv errors.jsonl errors.$(date +%Y%m).jsonl
gzip errors.2026*.jsonl

17.4 /changes 變動總表

Model: app/models/changes_summary.py. 路由: /changes（main.py）。

  • 彙整 last_viewed_at 以來的 timeline state_change + field_change，配合新增 issue / 關單。
  • 分類：重點項目（紅線以上 / 狀態回退 / Check-in 延期 / 新增 / 關單）vs 其他變動（正常進展、欄位編輯）。
  • 預設 include_own=True（含本人操作）。Navbar bell badge 用 count_important()，同樣計入所有人。
  • 有 ?node=<id> 篩選——Node 篩選只保留 cell 變動（field / new / closed 是 issue-level，不屬於特定 node 所以被排除）。
