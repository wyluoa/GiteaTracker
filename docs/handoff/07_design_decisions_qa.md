# 07 — 設計決策 Q&A

> 這份文件記錄專案討論過程中的關鍵決策與理由。如果你(接手的 AI 或開發者)覺得某個設計看似奇怪、想要「優化」或「改變」它,**請先讀這裡**,多數情況下你會發現這是經過討論後的 trade-off 決定。

## 技術棧選擇

### Q: 為什麼選 Flask 而不是 FastAPI?
**A**: 團隊環境的主要 Python 版本是 3.6.10(但專案主機也有 3.9/3.12/3.14),希望技術棧對開發者友善。Flask 學習曲線最平緩、生態最成熟、後端渲染不需要另外做 SPA 前端,對「內網工具 + 單人開發 + 日後可能交接」的情境最合適。FastAPI 的優勢(async、OpenAPI 自動產生)對內網低流量工具沒有明顯幫助。

### Q: 為什麼不用 React/Vue 做前端?
**A**: 同上原因。內網工具不需要 SPA 體驗,且 HTMX + Alpine.js 已經能達成「點 cell 開 side panel、改狀態不 reload 整頁、動態載入 timeline」這些互動需求。沒有 build step、不需要 npm、不需要另一個工程師會前端 JS framework,整體複雜度大幅降低。

### Q: 為什麼不用 ORM(Flask-SQLAlchemy)?
**A**: 沒有硬性規定,接手的開發者可以自行決定。使用者團隊已熟悉 SQLite,原生 sqlite3 + Row factory 對這個規模的專案已經夠用,也更透明。如果接手者偏好 ORM 可以加上,只要保持 schema 一致。

### Q: 為什麼選 Python 3.12 而不是 3.14?
**A**: 3.14 太新,部分套件可能還沒跟上。3.12 穩定已久、生態完整。3.9 也是選項但 2025-10 EOL,新專案不建議從這裡起步。

### Q: 為什麼存 SQLite 而不是 PostgreSQL?
**A**: 團隊已有用 SQLite 的內網服務經驗,硬碟層面的備份也已經由 OS 處理。300 題 × 9 node × 3 年的規模對 SQLite 來說非常輕鬆。部署時只有一個 .db 檔,無需額外管理 DB server。

### Q: 附件為什麼不存 DB?
**A**: 300 題 × 3 年 × 每題平均數張截圖,存 BLOB 會讓 .db 膨脹影響備份與查詢效能。存檔案系統後 DB 只存路徑,輕量、備份彈性大、nginx 直接 serve 也快。

## 權限與帳號

### Q: 為什麼不串公司 LDAP / SSO?
**A**: 使用者明確表示「想要獨立於 Gitea 另外做帳密驗證」。內網工具也不一定有 LDAP 管理者願意配合整合。自助註冊 + super user 審核已足夠低摩擦。Phase 7 如果有需求再說。

### Q: 為什麼註冊要 super user 審核?
**A**: 避免:
1. 非公司內部人員亂註冊看到內部題目
2. Email 是否真的公司內部無法自動驗證
3. 註冊時可以順便指派到適合的 group,一步到位

### Q: 為什麼 group 是「多對多」而不是「user 直接指派 node」?
**A**: 使用者明確說「我們目前組織架構會比較像是一個 group 可以對應多個 node」,且「一個 node 可以被多個 group 編輯」。這反映現實組織結構:通常是「QC team 管某幾個 node」而不是每個人各自被指派 node。Group 抽象化讓成員變動時只改 group membership,不用一個個 re-assign node。

### Q: 為什麼 Meeting Owner 是唯一能新增題目的人?
**A**: 使用者說「新題目是我建的」。會議前本來就要確認題目品質、合併重複、決定哪週討論,放寬新增權限反而會讓題目品質下降 + meeting owner 仍要花時間審核。未來想放寬隨時可以,反過來比較麻煩。

### Q: 為什麼 requestor 設計成雙欄位(`requestor_user_id` + `requestor_name`)?
**A**: 使用者問「requestor 連到 User 表或自由填上是好的做法嗎?」。兩者都有道理:
- 舊資料匯入時 requestor 多半只是文字,沒有對應 user
- 現代的 requestor 可能是系統 user
- 外部合作者不會是系統 user

雙欄位解:`requestor_name` 永遠是顯示來源(離職、改名都不影響歷史),`requestor_user_id` 提供未來「追蹤某人提過幾題」之類的功能。匯入舊資料時只填 `requestor_name`。

## 狀態與流程

### Q: 為什麼允許狀態任意切換,不強制順序?
**A**: 使用者說:「我們可能會在狀態之間切換,不一定會每次都按照順序(UAT → UAT done → 因為需求改動而又改回 UAT)」。真實工作流程就是會有回退,系統不能假設線性進程。

### Q: 為什麼 `Done` 和 `UAT done` 一定要分開?(不能合併成「完成」)
**A**: 使用者明確說:「我們的 Done 是程式已經上線,UAT done 是程式還沒上線,可能還是要有所區隔」。這個區分對規劃批次上線很重要。顏色也刻意設計成綠/藍兩色系徹底區分。

### Q: 為什麼 TBD 和 On Hold 是兩個不同層級的概念?
**A**:
- **TBD** 是 **cell 層級**的狀態(單一 node 未決定)
- **On Hold** 是 **issue 層級**的狀態(整題暫停)

使用者說 TBD 「比較像是不一定要測,可以直接做決定評估要不要上線或不上線,也有可能像等外部因素」。兩者都有「懸而未決」的味道,但:
- TBD 在某個 node 上等決定,其他 node 可能照常推進
- On Hold 整題都不動

強迫使用者合併會失去表達力。使用者接受加上 On Hold 作為 issue 層級狀態。

### Q: 為什麼紅線以上的 UAT / TBD 要特別顯示?
**A**: 使用者在現有 Excel 已經有「手動劃紅線 + 紅字標示 UAT/TBD」的做法,這是既有工作習慣。系統化時保留這個概念但讓紅線數位化、可設定。紅線以上代表「這題已經拖很久了,負責人該關心了」。

### Q: 為什麼紅線的位置是手動設定,不是自動算「超過 N 週」?
**A**: 使用者回答 (b):「還是維持『手動劃一條線,線以前都算舊題目』的概念,但讓這條線可以被設定」。手動設定讓 meeting owner 可以根據實際情況調整(例如放假、某段時間進度集中放緩)。

### Q: 為什麼紅線以上是整格色塊變紅,而不是 cell 整格紅底?
**A**: 使用者原本選了整格紅底,後來看到草圖覺得太醒目、破壞版面一致性,改口「像 UAT 那樣但改用紅色就好」。最終是色塊變紅 `#c0392b`,cell 背景保持白色。

### Q: 為什麼關單要手動,不是自動?
**A**: 使用者說:「在這個系統上可以顯示可以 closed,並且讓我手動處理可以讓這題移到另一 closed 的分頁,這個 closed 的權限只有我或是其他指定的 super user 可以執行」。自動關單會讓 Meeting Owner 失去最終把關的機會。

## 資料設計

### Q: 為什麼 `issues` 表有 `latest_update_at` 和 `all_nodes_done` 這些衍生欄位?
**A**: 效能。主畫面列表需要顯示這些資訊,如果每次都 JOIN + GROUP BY 計算,300 題會變慢。寫入時多花一點力氣維護 cache,讀取時快得多。

### Q: 為什麼 timeline / state 要存 `*_name_snapshot`?
**A**: 使用者希望「離職或轉組的人名可以維持他原本的名稱」。user_id 負責連結,name_snapshot 負責顯示。即使帳號後來被 disable 或 display_name 被改,歷史紀錄仍然顯示當時名字。這是稽核完整性的基本要求。

### Q: 為什麼 node 要有固定 `code` 又要有 `display_name`?
**A**: 使用者說「Node 還有可能會增加,狀態通常為這幾種」。未來可能改名、可能合併、可能停用。固定 `code` 當外鍵穩定來源,`display_name` 讓管理員改顯示名稱不破壞歷史資料。

### Q: 為什麼刪除用 soft delete?
**A**: Audit 需求。任何真的刪除會破壞歷史連結和統計數字。`is_deleted = 1` 讓資料仍在但不在 UI 出現,必要時可以 unarchive。

### Q: 為什麼 `short_note` 跟 timeline 要並存?
**A**:
- **short_note** 放「持續性的現況描述」:例如「等廠商 3/15 回覆」、「待 spec 確認」。會持續顯示在 cell 下方。讀的人看一眼就知道目前狀況。
- **timeline** 放「一次性的事件」:狀態變更、討論補充、會議記錄。

兩者互補。如果只有 timeline,使用者得點進去翻歷史才知道「為什麼這 cell 卡住」。如果只有 short_note,過往紀錄會消失。

### Q: 為什麼留言和狀態變更放同一個 timeline 表?
**A**: 使用者問「這個表格應該不會有一版討論留言,我想確認你所說的一般討論留言是什麼?」。釐清後結論:所有跟題目相關的「事件」都放同一個 timeline,用 `entry_type` 區分類型,用不同樣式顯示。這樣查詢簡單(一個 query 搞定整題歷史)、篩選彈性(UI 可選「只看狀態變更」等)。

## 會議與工作流程

### Q: 為什麼要有「會議模式」這個獨立頁面?
**A**: 使用者選了「兩者都做」(一般用 side panel 個別記,會議當下用連續記)。一般情況下在 side panel 裡記會議紀錄很自然,但開會時切換十幾題太慢,需要一個連續記錄的介面。兩種路徑都能產生 `meeting_note` 類型的 timeline entry。

### Q: 為什麼 Summary Dashboard 要把「紅線以上未完成」放大字紅色?
**A**: 使用者說「Summary Dashboard 想要紅線以上的再著重看(字體較大且紅字)」。這是目前最痛的點——知道還有多少「老題目」卡在自己 node 上。Dashboard 第一眼就要看到,不需要進主畫面再篩選。

### Q: 為什麼要有上線行事曆?
**A**: 使用者說「也需要能夠一眼看出哪些題目卡很久,每個 node 剩下的題目有幾題」以及「我們為了開發方便,通常都是一次累積幾題一起上線」。行事曆讓 Meeting Owner 看到「下週三有 5 個 node 要上線」,方便規劃批次上線。快速 ✓ 按鈕讓批次上線完可以在同一頁快速勾掉,不用一題一題開 side panel。

### Q: 為什麼 Closed 分頁要支援 reopen?
**A**: 使用者說「反關單有可能會需要,但非常少見」,並要求「要留紀錄」。少見不等於不需要,越少見的操作越需要留痕(因此 reopen 要填理由並寫 audit log)。

### Q: 為什麼要有「版本差異黃底」?
**A**: 使用者描述現有 Excel 的做法:「每個版本的狀態更改都會用黃色醒目底來標示,讓大家知道這版跟上一版差異在哪」。系統化後改成 per-user 「自上次查看以來的變動」,比原本的版本差異更精準(每個人看到的都是「對他而言的新變動」)。

## UI 與視覺

### Q: 為什麼顏色要這樣配?
**A**: 經過多次調整。最終版:

- 綠色家族 = 「好結果」(Done 深綠 + UAT done 本來是淺綠,後改藍)
- 但使用者堅持 Done 和 UAT done 必須視覺上明顯區分(因為業務語意真的不同),所以 UAT done 從「淺綠」改到「藍色」,完全不同色系
- 橘色 = 「進行中要測」(UAT)
- 灰色 = 「開發中」(Dev)
- 紫色 = 「未決」(TBD)
- 淺灰 = 「不需處理」(Unneeded)
- 紅色 = 「警示」(紅線以上的 UAT/TBD)

### Q: 為什麼排序是舊題目在上、新題目在下?
**A**: 使用者說「週次分組應該由舊到新會比較好(上面舊的,下面新的),因為新題目有可能還需要時間開發,舊題目會需要讓使用者看到需要解決」。違反多數網站的「新的在上」習慣,但符合這個 domain 的需求:舊題目才是問題。

### Q: 為什麼 cell 編輯要用 side panel 而不是 inline?
**A**: 使用者說「點開一個 side panel / modal 再改應該可以,因為這也不需要很常改動」。Inline 編輯快但誤觸風險高,且 side panel 可以同時顯示 timeline 提供上下文,是比較「有意識」的操作。cell 改動有 audit 意義,值得慢一點。

### Q: 為什麼主畫面要顯示 check-in 日期(不藏在 tooltip)?
**A**: 使用者說「版面上的 cell 可能會需要顯示 check in 日期,讓我方便去看什麼時候要上線,或是你有其他方式可以讓我方便整理每一題未來要在什麼時候上線」。這個資訊太常用,藏起來會逼使用者一直開 side panel。其他細節(最後更新時間等)則放到 side panel 不顯示。

### Q: 為什麼新系統不做相鄰同狀態的視覺合併?
**A**: 使用者最終說「新系統就不要再做合併儲存格的處理了」。原因:
- 有了顏色區分,相鄰同色已經夠清楚
- 合併後點擊要額外判斷「點到哪個 node」會複雜
- 批次操作功能可以一次改多個同類型,更方便

## 匯入舊資料

### Q: 為什麼歷史資料全部掛在 `legacy` 假帳號?
**A**: 使用者說「舊資料可以先都掛在 legacy 假帳號,已經 closed 的題目也要一起匯進來」。Excel 沒有「誰改的」資訊,硬要回推是不可能的。legacy 帳號讓歷史資料的 `updated_by_user_id` 有值(不 null),查詢邏輯統一。

### Q: 為什麼要處理 Excel 合併儲存格?
**A**: 使用者說「我目前的 Excel 還有合併儲存格,通常只有同一列的有合併,在匯入的時候可能要考慮這情況」。`openpyxl` 讀合併儲存格時,只有左上角那格有值,其他是 None。匯入前要掃 `ws.merged_cells.ranges` 把值展開。

### Q: 為什麼匯入腳本要 idempotent?
**A**: 使用者同意「並行期間 Excel 仍是 single source of truth」,這意味著在正式切換前,可能會多次從 Excel 重新匯入到新系統覆蓋。腳本必須可以重複執行而不會重複插入或搞壞資料(UPDATE by `display_number`)。

## 效能

### Q: 為什麼要做 virtual scrolling?
**A**: 300 題 × 9 node = 2700 cell,加上 timeline 資訊、版本差異黃底計算,一次全部 render 會慢。Virtual scrolling 只 render 可視範圍,滑動時動態載入。**但**這是 Phase 6 才視實測決定是否需要的優化,Phase 1~5 先不做。

### Q: 為什麼主畫面預設不載入 Closed?
**A**: 已關單的題目不會再被動到,載入只會拖慢主畫面。獨立成 Closed 分頁,預設只載最近 N 筆,要看舊的要搜尋或翻頁。

### Q: 為什麼 session 逾時 1 天不是更短?
**A**: 使用者回答「Session 逾時一天登出」。內網工具 + 低敏感度,1 天平衡安全與使用體驗。

## 「等等我想到一件事」反覆釐清過的重要議題

### 一個 node 可能屬於多個 group?
是。使用者說「一個 node 可以被多個 group 編輯」。DB schema 的 `group_nodes` 表是多對多。

### 一個人可以屬於多個 group?
是。使用者說「一個人可能會負責多個 node」。DB schema 的 `user_groups` 表是多對多。

### 密碼忘記是自助 reset 還是管理員重設?
自助 reset。使用者選「密碼忘記走自助 reset」。需要 SMTP 設定。

### 權限檢查的顆粒度是到 cell 嗎?
到 node 層級。檢查「當前 user 是否屬於任一 group,且該 group 能編輯這個 node」。同一個 user 在不同 issue 的同一個 node 權限都是一樣的(不會「只能編輯某些 issue 的這個 node」)。

### Meeting note 要有會議週次欄位,跟 issue 的 week 有關係嗎?
無直接關係。`timeline_entries.meeting_week_year` / `meeting_week_number` 是該筆 meeting note 發生在哪一週的會議,issue 的 week 是題目被首次討論的那一週。兩者可能相同,也可能不同(例如某題在 wk320 首次討論,wk325 又被討論就會有 wk325 meeting note)。

### Audit log 和 timeline 的差別?
- **timeline** 是**題目層級**的歷史,給任何看題目的人讀(包含一般 user)
- **audit_log** 是**系統層級**的操作紀錄,只有 super user 能看,包含帳號管理、group 變動、紅線調整、關單、reopen、刪除等敏感操作

有些操作會同時寫兩邊(例如關單會在 timeline 寫一筆給題目看的人,在 audit_log 寫一筆給 super user 回溯)。
