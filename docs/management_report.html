Gitea Tracker

以 AI 輔助開發打造的內部 QC Script 追蹤系統 — 把會議管理從 Excel 升級到可審計、可協作、可視化的 Web 平台

Report Version 2026-04-21  ·  Prepared for Management Review

執行摘要

Gitea Tracker 是一套取代原本 Excel 控管流程的內部網頁系統， 用於追蹤約 300 筆 QC Script Request 跨 10 個工作階段（Node）的進度。 由單一開發者以 AI 輔助開發模式打造， 在約一週內交付完整可上線的 Phase 0 ~ Phase 6 功能， 至今歷經三輪公司內部 UAT 反饋迭代與一輪維運基礎建設強化（2026-04-21 引入正式的 DB migration 系統與 AI 協作安全機制）。

1. 為什麼需要這套系統

1.1 原本的痛點：Excel 控管的結構性限制

議題Excel 控管Gitea Tracker
同時編輯 檔案被鎖死、同仁互搶編輯權、改動常覆蓋彼此 多人即時編輯，每格獨立鎖、變更不衝突
變更追溯 無法得知誰 / 何時 / 改了什麼 所有狀態改動、留言、附件全存進 timeline + audit log
權限控管 打開即可改，無法限制單人只能改自己的 Node Group 制分工權限；Done 僅 super user、Unneeded 限 manager 以上
搜尋與篩選 Ctrl+F 僅能單檔搜尋，無法跨條件組合 跨題號 / Topic / JIRA 多欄位搜尋，支援 AND / OR 進階篩選
儀表板與趨勢 每週人工整理 pivot、每季人工算關單率 Dashboard 即時卡片 + 趨勢圖，可切 YTD / 30 週 / 50 週
通知機制 改動不通知，同仁要主動問 Email 通知新帳號申請、狀態異動、忘記密碼
行事曆視角 無，需要另外維護 月曆呈現 Check-in 日期，附「可上線清單」主動揪出逾期項目
資料備份 一份 .xlsx 壞了或誤刪就沒了 單檔 SQLite + 附件目錄，每日自動打包

核心問題不在 Excel 本身，而是追蹤與協作的本質已超過試算表能承載的複雜度。 每週 QC Meeting 需要整合 10 個 Node 的進度、回溯歷史決策、提醒 Red Line 逾期項， 在 Excel 上每項都要額外人工作業，容易漏、容易錯、也難交接。

1.2 不直接用現成 Gitea Issue / Jira 的原因

公司既有 Gitea 是 issue tracker、適合單線程開發追蹤； 而這裡的場景是同一題跨多個 node 的並行進度矩陣—— 每題要同時顯示 10 個 node 的狀態、以週為分組、以 Red Line 為時間錨點。 這是 spreadsheet-like 的二維結構，現成 issue tracker 不擅長。 Gitea Tracker 填補這個空白，同時保留與 Gitea 的連動（題號可 hyperlink 直達 Gitea issue）。

2. 系統功能亮點

◆

多狀態矩陣追蹤

每題跨 10 個 Node 的狀態（Done / UAT / TBD / Dev / Unneeded）， 以色塊 chip 呈現、Red Line 逾期自動變紅警示。

☵

三主題 UI

Light / Dark / Gray 三鍵循環。Gray mode 全頁灰階， 只保留 UAT / TBD 紅色，特別適合會議投影與列印。

▣

Dashboard 即時統計

Ongoing / Ready to Close / 匯入建議關題 / Closing Rate 等卡片即時計算；Closing Rate 支援 MtM 含與不含兩種定義。

◎

洞察分析

Almost Done（剩 1-2 node）、Stale Topics（90 天未動）、 Bottleneck（各 node 卡關數）、Per-Node Stats，輔助週會決策。

≡

進階 AND / OR 篩選

一般篩選 + 最多 3 組「Node + 狀態」條件，可切換 AND / OR 邏輯。例如「A10 Done 且 A12 UAT」或「任一 node 卡在 TBD」。

⇅

Excel 雙向互通

支援舊 Excel 格式上傳（帶 diff preview 人工確認）， 也能匯出為同格式 Excel。過渡期同仁可隨時回退。

⧉

完整 timeline

每一格改動、comment、meeting note、附件皆入 timeline； 超級使用者有 audit log 可跨帳號追查任何異動。

✉

通知與行事曆

SMTP email 通知新帳號申請 / 密碼重設；Calendar 月曆提供 Check-in 時間軸與「可上線清單」主動提醒逾期。

◐

欄位級版本差異（2026-04-21）

Topic / Owner / JIRA / UAT Path 各自獨立 timestamp，只亮真正被改的那欄。 讓使用者快速鎖定「這次改了什麼」而非「這列有東西改」。

⚙

DB Migration 基礎建設（2026-04-21）

引入正式的 schema version tracking 與一鍵升版流程， 含 backup / rollback SOP。系統長期維護的技術債自此有基礎。

3. AI 輔助開發方法論

這套系統最值得分享的不只是功能本身，更在於開發過程採用 AI 輔助模式—— 由團隊成員主導需求與決策，Claude（Anthropic 的 Claude Code CLI + Claude Opus 4.7 模型） 承擔程式碼撰寫、重構、測試規劃等執行層面工作。

3.1 為什麼這是「好的 AI 嘗試」

① 人類主導、AI 執行

所有功能範圍、UX 決策、驗收標準皆由開發者（Meeting Owner）定義。 AI 不替開發者做架構決定、也不自動上版； 每個 commit 都經過開發者 review 才推上版庫。這確保系統行為是業務需求驅動， 而非 AI 臆測。

② 可驗證的產出

所有程式碼公開於內部 Git、每個 commit 都有完整訊息與對應的功能說明， 後續同仁可以 git log 逐條追溯每一個變更的動機與實作。 Git 歷史本身就是 AI 協作透明度的佐證。

③ 快速迭代 + 高品質

公司內部 UAT 提出回饋後，從反饋到部署的週期從「週」縮短到「小時」。 Tracker 標題凍結、Dashboard 題號展開、進階篩選 AND / OR、 多行 UAT Path、公司 Chrome 快取問題——這些都是 UAT 回饋當日或隔日即交付。

④ 零外部授權成本

整個技術堆疊（Python / Flask / HTMX / Alpine / SQLite / Tabler）皆為開源免費， 不需任何軟體授權支出；在公司內部伺服器 user-space 執行，也不需 sudo 或新開機器。

AI 輔助開發的真正價值，不在於取代工程師的判斷， 而在於把一位有產品 sense 的同仁的產出能力，放大到接近小型產品團隊的規模。

5. 技術決策與可維護性

5.1 為何選擇這個技術堆疊

層選擇對管理上的意義
後端：Python + Flask 公司標配、人人能改 未來接手不需額外訓練，招募 / 交接門檻低。
資料庫：SQLite 單一 .db 檔 不需要額外 DB server，備份就是複製檔案，零運維成本。
前端：HTMX + Alpine 沒有 build step 不需要 webpack / Node.js 生態；改一個檔直接生效，上手快。
CSS：Tabler (CDN) 免費、內建 UI 元件 視覺專業感即時達成，不需要 UI 設計師。
部署：純 Python + nohup 無需 sudo 任一 Linux 帳號都能起服務，不需 IT 開 ticket 裝 service。

5.2 維護性設計

  • 正式 DB migration 系統（2026-04-21 導入）： migrate.py runner + schema_version 追蹤 + 獨立 migration 檔案 + 完整 SOP（deploy/MIGRATION_SOP.md）。每個 migration 必須冪等， 同步更新 schema.sql；一鍵升版 deploy/migrate.sh 自動備份 → 停機 → git pull → migrate → 啟動。這補齊了系統長期維護的關鍵基礎設施。
  • 完整 audit log：任何 admin 動作都留紀錄，責任可追溯。
  • Swagger API 文件：每個 route 有 docstring 自動生成 API 說明頁， 後續接手者看 /apidocs 即可理解系統。
  • Cache-bust 機制：CSS 以 file mtime 作為 fingerprint 加在 URL， 避免企業代理快取讓新功能無法生效，部署即刻生效無需提醒同仁清快取。
  • Soft delete + Reopen：誤刪可救、誤關可重開，降低人為失誤成本。
  • Excel 雙向互通：過渡期若需切回 Excel 亦可匯出，風險可控。
  • AI 協作安全機制（2026-04-21 導入）： .claude/hooks/ PreToolUse 攔截器保護 data/attachments/ 不被 AI 改寫、migrate.py 執行前強制檢查近 24h 備份； CLAUDE.md 記錄架構要點、已決定事項、遵守規則，讓下一位 AI / 新同仁 有清晰的交接文件。
  • in-app 文件頁（2026-04-21 導入）： 使用手冊 / 開發者指南 / 管理報告皆可直接從導覽列「說明」下拉存取，無需額外文件伺服器或共用雲端檔。

5.3 安全與合規

  • 密碼以 bcrypt 雜湊存放，不可還原。
  • Session 以 Flask 簽名 cookie 保護，SECRET_KEY 寫在 .env 不入版控。
  • 所有 DB 查詢以 parameterized query，免 SQL injection。
  • 檔案上傳限制格式（PNG / JPG / PDF）與大小（5 MB × 最多 3 個）。
  • Admin 動作進 audit log，需要時可追溯。

6. 對團隊與管理層的具體效益

對會議主持人（Meeting Owner） 週會前不需再手動整理 Excel pivot；Dashboard 打開即看到 Red Line 逾期、Ready to Close、 Almost Done、Stale Topics 四張清單，直接依此訂會議重點。 原本需要半天準備週會的時間壓縮到幾分鐘。
對 Node Owner（實際 UAT 執行者） 不再需要搶編 Excel；登入後只看自己 group 負責的 node，所有改動被即時 timeline 記錄。 Calendar 的「可上線清單」會主動提醒有哪些 check-in date 到期未完成。
對管理層 隨時從 Dashboard 取得客觀數字：Ongoing / Closed 比、Closing Rate 趨勢、 Per-Node Bottleneck。彙報不再依賴「Meeting Owner 週會報告」這個單一管道。 資料可信度高、可溯源（每個數字都可以點進去看明細題目）。
對 IT / 資訊安全 完整 audit trail、soft delete、bcrypt 密碼、email 忘記密碼流程，符合基本資安要求。 系統跑在使用者自己的 Linux 帳號下，不需額外開機器、不動公司網路設定。

7. 對部門 / 公司的戰略意義

7.1 AI 實戰經驗累積

這個專案是部門第一個完整導入 AI 輔助開發的產出，沉澱下來的價值包括：

  • 團隊可複製的 AI 使用模式：從「人類定義需求 → AI 實作 → 人類 review 並測試」 的三段式流程，後續其他同仁要做類似的工具時，有具體參考。
  • 對 AI 能力邊界的第一手認識：團隊親身體驗 AI 可做什麼（寫 CRUD、 debug CSS、排版建議、文件撰寫）、不可做什麼（做產品決策、選技術堆疊、決定驗收標準）。
  • 工具鏈建立：Claude Code CLI 的使用流程、prompt 寫法、 Git commit 習慣、透過 memory 檔把專案上下文保留下來等。

7.2 為部門建立「內部工具自製」的文化

長期來看，由團隊成員自己打造所需工具，而非依賴外購軟體或外部委託：

  • 更貼近實際流程：工具是為了「這次會議」「這群人」設計的，不需妥協。
  • 快速迭代：需求變就改，小時級交付而非月級。
  • 知識留在公司：程式碼、設計決策、使用經驗全部內部累積。

7.3 可擴展的模式

相同的開發模式可延伸到其他內部需求：

  • 其他部門的流程管理工具（如設備借還、採購簽核追蹤）
  • 自動化報表平台
  • 內部知識庫 / FAQ 系統

每個新工具都能重用 Gitea Tracker 已驗證的架構（Flask + SQLite + HTMX）， 加速後續產出。

8. 持續演進與風險管控

8.1 已規劃的下一階段

  • 更細的權限：Node owner 可設代理人、休假不延誤進度。
  • 與 Gitea issue 雙向同步：關單動作同步標註 Gitea issue。
  • API 對外：允許其他內部系統查詢 tracker 狀態（例如報表服務、ChatOps）。
  • 更深的 AI 整合：依歷史資料預測 node 延誤風險、自動生成 weekly summary。
  • 自動化 weekly reminder email：Red Line 逾期項目 / Stale Topics 每週發信給相關 Owner。

8.2 潛在風險與緩解措施

風險緩解措施
單一開發者依賴 程式碼與文件完整公開於內部 Git；開發者指南（Developer Guide） 將系統所有決策記錄下來，任何 Python 基礎的同仁可接手。 2026-04-21 新增 CLAUDE.md，把「新人 / 新 AI 需要知道但從程式碼看不出來」的 慣例（例如 venv 路徑、port 實際值、已決定不要再討論的事項、Cache 欄位語意）集中記錄， 降低接手門檻。
SQLite 擴充性 目前 300 題、10 人使用遠低於 SQLite 瓶頸；若未來規模成長， 可遷移至 PostgreSQL（Flask SQL layer 已抽象化）。
Schema 演進風險 2026-04-21 導入正式 migration 系統解決此風險。 每次 schema 改動都是獨立檔案 + 冪等性檢查 + 追蹤歷史， 升版前有 --dry-run 預覽、失敗可 rollback 到前一個備份。 新增 migration 有 SOP 與 checklist，不靠 ad-hoc ALTER TABLE。
AI 誤動資料的風險 2026-04-21 導入 .claude/hooks/ PreToolUse 攔截器： AI 欲寫入 data/attachments/ 或執行 migrate.py（非 dry-run） 且 24h 內無備份時自動擋下並提示正確流程。 Skills 中對 DB 有破壞性的流程（/run-migration）標記為 「僅人類可觸發」（disable-model-invocation: true），AI 不會自作主張。
瀏覽器環境差異 已處理公司代理 CSS 快取問題；持續以公司 Chrome 為首要 QA 環境。
同仁抗拒改用 保留 Excel 雙向互通作為後路；漸進式移轉（先 read-only 開放、 再逐步開放編輯）；User Guide 有繁中詳細說明，且直接可從導覽列「說明」開啟，無需另外找連結。
